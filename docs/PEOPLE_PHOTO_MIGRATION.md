# Private People-Photo Migration

<!-- AGENT: DEVOPS -->

## Target Model

New people photos are stored in the private `people-photos` Supabase Storage
bucket. The database stores only `people.photo_path` in this format:

```text
{church_id}/{person_id}/{generated_uuid}.webp
```

Signed URLs are generated by authenticated server routes and expire after 10
minutes. Do not persist signed URLs.

## Inventory

Run the count-only inventory as `service_role` after migration 039:

```sql
SELECT
  church_id,
  church_display_id,
  church_name,
  category,
  people_count
FROM public.people_photo_reference_inventory
ORDER BY church_name, category;
```

The view is intentionally unavailable to `anon` and ordinary `authenticated`
roles and does not expose raw object paths or public URLs.

Categories:

- `private_photo_path`: already using the private path column.
- `empty`: no photo reference.
- `legacy_public_people_photos_url`: old Supabase public people-photo URL.
- `legacy_storage_path`: old path-like value in `photo_url`.
- `external_url`: non-Supabase HTTP(S) URL.
- `malformed_or_unknown`: value that needs manual review.

## Tenant Pilot

1. Pick one tenant with a small number of legacy records.
2. Copy or reprocess each confirmed legacy object into
   `{church_id}/{person_id}/{generated_uuid}.webp`.
3. Verify the copied object opens through the admin signed-photo route.
4. Update that row to set `photo_path` and clear `photo_url`.
5. Confirm list, profile, and portal own-photo views still load.
6. Repeat for the rest of the tenant before moving to another tenant.

Do not delete legacy objects until the tenant has passed verification and the
rollback window has expired.

## Unsupported Legacy Values

External URLs, malformed values, missing objects, and cross-tenant-looking paths
must be reviewed manually. Do not silently delete or rewrite them.

If an existing object cannot be safely decoded and re-encoded, leave the record
unchanged and request a fresh upload from an authorized administrator.

## Rollback

Migration 039 is additive for the database. `people.photo_url` remains intact
during rollout, and old application code can ignore `photo_path`.

If the private-photo deployment fails:

1. Roll back application code.
2. Leave private objects in storage.
3. Do not clear `photo_path` values unless a forward fix requires it.
4. Avoid reopening public bucket access unless the privacy tradeoff is
   explicitly approved.

Remove legacy fallback support only in a later PR after every tenant is
inventoried, copied or reprocessed, verified, and monitored.
